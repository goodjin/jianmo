# Backlinks：大仓库性能与截断策略（M333）

## 背景

Backlinks 在“大仓库 + 大量 Markdown 文件”场景下，最容易出现：

- 初始化扫描慢
- 频繁刷新导致卡顿
- 结果过多导致 UI 不可用

## 结论（策略优先）

本阶段先固化 **截断与降级策略**，不强行做“全量准确的 backlinks”。

## 策略

- **搜索范围**：
  - 默认：当前 workspace
  - 排除：`node_modules/`、`dist/`、`.git/` 等明显无关目录
- **增量与节流**：
  - 文档切换/输入后，延迟计算（debounce）
  - 最多每 N 秒更新一次（throttle）
- **结果截断**：
  - 最多展示前 **200** 条（可配置）
  - 超出时显示“已截断”提示，并引导用户用 VS Code 全局搜索
- **失败/超时降级**：
  - 超过时间预算（如 200ms）直接返回“部分结果”，并在诊断里记录耗时
