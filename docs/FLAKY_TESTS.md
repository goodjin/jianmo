# Flaky 测试管控（M245）

目标：让 CI 的红灯尽量代表真实回归；对偶发失败有统一处理办法，避免“重跑直到绿”。

## 1. 发现 flaky 的判定

- 同一 commit / 同一环境下，重复运行测试出现 **随机失败**。
- 失败与时间/并发/动画帧/网络/系统负载相关。

## 2. 首选修复顺序（从好到坏）

1. **去掉时序赌运气**：用确定性的 flush（`nextTick` / `requestAnimationFrame` / 假时钟）或更明确的状态断言。
2. **隔离外部依赖**：把网络/文件系统/时间抽象为可注入依赖（mock）。
3. **缩小断言范围**：只测行为结果，不测不稳定的中间状态或 DOM 细节。
4. **拆分用例**：把一个巨大用例拆成多个小用例，定位更准，也更不易超时。

## 3. 临时止血策略（不推荐长期使用）

- 禁止在 CI 层“盲目重试整套测试”当作常态（会掩盖真实回归）。
- 如果必须重试：必须把 flaky 项**记录到清单**并安排修复时间窗。

## 4. 建议的产物

- 新增回归用例时，优先用 Vitest（`npm test`），其次才是 UI e2e。
- 对 UI/异步强的测试，优先写“稳定 flush helper”（例如本仓库 `OutlinePanel` 的筛选防抖就需要显式等待）。
