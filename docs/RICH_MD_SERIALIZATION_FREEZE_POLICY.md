# Rich ↔ Markdown 序列化冻结策略（M286）

> 目标：把“哪些差异是可接受的规范化、哪些差异必须视为 bug”写成统一口径，并用测试夹具持续监控。

## 1. 我们冻结什么

- **用户语义冻结**：用户在 Rich 中编辑后保存，Markdown 的“意义”不应发生变化（标题层级、列表层级、表格结构、链接目标、代码内容、公式内容等）。
- **格式规范化允许**：在不改变语义的前提下，允许少量“排版规范化”（见下节）。
- **导出一致性**：若 Rich 保存后 Markdown 发生规范化，PDF/HTML 导出应与该规范化后的 Markdown 一致，避免“看着对、导出错”的分叉。

## 2. 允许的差异（不算 bug）

以下差异允许发生，但必须满足“语义不变”：

- **空白规范化**：行尾空格、空行数量的轻微变化（不影响段落分隔语义）。
- **表格对齐行规范化**：如 `| --- |` 对齐行的空格、对齐符号格式被统一（但列数/单元格内容不丢失）。
- **列表 marker 规范化**：`-`/`*`/`+` 等 marker 的统一（但层级与任务状态不变）。
- **引用/分割线写法统一**：同一语义的等价写法被序列化为一种稳定形式。

## 3. 必须视为 bug 的差异

以下任一发生即视为 bug（需要修复或明确声明为限制）：

- **结构变化**：表格列数改变、单元格内容被拆行/合并导致语义变化；列表层级或有序编号语义变化；标题层级变化。
- **内容丢失/改写**：代码块/行内代码内容被改写（尤其包含 `|`、反引号、转义符）；链接 URL 改变；图片引用丢失。
- **块边界破坏**：把一段变成两段或反之，导致渲染/导出含义变化（除非属于允许的空白规范化且渲染等价可证明）。

## 4. 工程门禁与证据

- fixture 回归：`webview/src/__tests__/richFixtureRoundTrip.test.ts`
  - **优先策略**：第一次 normalize 后应稳定（P→S→P→S 的二次输出与一次输出一致）。
  - **例外**：若上游序列化器存在已知不可控规范化（例如表格对齐行），可以对个别 fixture 做“弱等价”断言，但必须：
    - 解释原因
    - 增加“关键内容不丢失”的断言（例如 `a|b` 在同一单元格内仍存在）
- 关键语义回归：已有专项测试（如换行、列表缩进、表格粘贴与键盘行为）必须持续通过。

## 5. 用户口径（支持/排障）

- 用户如反馈“保存后格式变了”：先判断是否属于 **允许的规范化**；若不属于，按 bug 处理并要求提供“复制诊断”与最小复现 Markdown。
