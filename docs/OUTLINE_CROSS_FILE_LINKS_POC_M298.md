# 大纲：跨文件链接（可选）产品决策与 PoC（M298）

## 结论（本阶段）

**不做 PoC**，只做产品决策与边界澄清：跨文件大纲/链接属于“知识库级功能”，会显著扩大范围并与 Backlinks/Workspace 索引耦合（见 M333/M334/M339）。建议延后到有全局索引与性能策略后再做。

## 需求拆解

用户想要的“跨文件链接”通常包含两类：

- **A. 大纲面板里显示其它文件的标题**（跨文件 Outline）
- **B. 大纲项点击后能跳到其它文件/位置**（跨文件 Jump）

两者都依赖：工作区扫描、缓存、增量更新、以及“目标位置可稳定定位”的锚点协议。

## 可行方案（仅评估）

### 方案 1：最小实现（仅支持 Markdown 链接跳转）

- 从当前文档中解析形如 `[text](./other.md#heading)` 的链接
- 展示为 outline 的“外部引用”节点
- 点击时让 extension 打开目标文件，并滚动到对应 heading slug

风险：

- slug 规则不统一（GitHub vs VS Code markdown vs Markly 的 slug）
- 文件变更后 hash 定位脆弱

### 方案 2：引入 workspace 索引（更正确）

- extension 侧维护 workspace Markdown 索引：标题列表、slug、from/to、更新时间
- webview 请求某个文件的 outline 或跨文件目标
- 引入性能预算与截断策略（与 M333/M334/M312 绑定）

## 本阶段拒绝理由（why no）

- **范围膨胀**：跨文件 = 索引系统，基本等同“知识库特性”。
- **性能与可靠性**：大仓库/多 workspace folder/Remote 场景风险高。
- **需要协议**：slug/锚点规范必须先定（否则只会制造新的“跳错”问题）。

## 未来落地前置条件（建议）

- 明确锚点规范（heading slug 规则与冲突策略）
- 工作区索引与性能策略（M333/M334/M312/M339）
- 兼容策略：找不到目标时的降级（打开文件 + 搜索 heading 文本）
