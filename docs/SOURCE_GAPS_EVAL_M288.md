# Source 补齐评估（M288）

> 目标：在“Rich 为主、Source 兜底”的策略下，列出 **Rich 当前覆盖不了但用户仍可能需要** 的编辑动作，并给出 Source 是否需要补命令/补 UX 的结论。

## 结论（当前版本）

- **Source 的定位**：可靠兜底（可编辑、可保存、可导出、可诊断），不追求 WYSIWYG。
- **补齐优先级**：只补“会阻塞写作/协作/导出”的动作；其余用文档明确“不支持/建议用 Rich”。

## 需要 Source 保证的能力（建议保持为 must-have）

- 基础编辑：输入、撤销/重做、查找/替换、复制粘贴
- Markdown 结构编辑：标题/列表/引用/代码块/链接/图片 的插入与最小快捷键支持
- 诊断/自救：Rich 失败时可一键降级到 Source（当前已实现）

## Rich 覆盖不完全、但 Source 可以作为兜底的点

- **复杂代码写作**：大量代码块、对齐、快速多行编辑（Source 更适合）
- **与其它扩展协作**：lint/formatter/片段等更偏向 TextEditor 的生态（Source 兼容性更好）
- **极端大文档**：Rich 可能触发性能/初始化失败；Source 应保证基本可用

## 候选补齐项（可选增强）

以下项如收到用户反馈再进入工程里程碑（否则仅文档说明）：

- Vim：是否引入 CodeMirror Vim 模式（见 M289）
- 多光标：CodeMirror 原生支持较好，但需确认与我们自定义 keymap/命令不冲突
- 格式化辅助：`tidyTables` / `fixMarkdown` 等命令在 Source 侧的入口与提示

## 不建议在 Source 做的事

- IR 类装饰器（已在中期移除）
- 表格格内 UI（Rich 专属能力）
