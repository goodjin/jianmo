## M9：Rich 正确性与编辑一致性（Stability + Fidelity）

> 状态：执行中  
> 目标版本：`1.5.1`（PATCH）或 `1.6.0`（MINOR，若新增配置/诊断入口）

### 1) 背景与问题

M8 解决了「大文档可用」与「启动更快」的问题。下一阶段要把 Rich 做到更“可信”：

- Rich ↔ Markdown 的序列化/解析在常见语法上不漂移（或漂移可控、有白名单）
- Rich/Source 频繁切换不引发内容跳变、光标/滚动错位、或竞态导致降级
- 出现问题时用户能“一键复制诊断信息”帮助定位

### 2) 统一验收口径（门禁与阈值）

#### 2.1 夹具（fixtures）覆盖面阈值

`docs/fixtures/m9/` 需覆盖至少以下类别（每类至少 1 个文件）：

- **表格**：含表头、对齐、合并/拆分后保存结果（若无法序列化为合并单元格则要明确策略）
- **列表**：嵌套列表 + 任务列表
- **脚注**（当前已启用）
- **代码块**：多语言（含未支持语言）+ 长代码
- **Mermaid**：多块 + 长图
- **数学**：行内/块级
- **链接/图片**：相对路径与带查询参数
- **引用块**：多层嵌套
- **超长行**：用于验证 wrap/scroll 策略不影响保存

#### 2.2 Round-trip 稳定性阈值

对每个 fixture，默认要求：

- **解析→序列化（P→S）**：`serialize(parse(md))` 与原 `md` **完全一致**  
  - 允许“规范化差异”的例外必须进入白名单（例如尾随空行、标题 ID 自动补全等），并在文档中逐条记录。
- **序列化→解析（S→P）**：`parse(serialize(doc))` 不应抛错；节点结构变化需可解释且有断言。

> 注：这里的“完全一致”以 **Markly 当前策略**为准；如果我们决定统一格式（例如表格对齐空格、列表缩进），则改为“规范化后等价”，但必须写清 `normalize()` 规则与覆盖测试。

#### 2.3 模式切换稳定性阈值（E2E）

- 对同一文档执行 **Rich↔Source 循环切换 10 次**：
  - `content` 不漂移（或漂移符合白名单规则）
  - 不出现 `rich-fallback-banner`
  - 光标位置变化在可接受范围（至少不跳到文档开头/末尾）

### 4) 风险与回滚

- **序列化漂移**：短期用白名单兜底，但必须把漂移原因写入 fixture README；长期再统一格式策略
- **E2E 不稳定**：优先补强 `__marklyE2E` 的稳定 API（必要时新增 hooks），避免依赖脆弱的 DOM 操作
