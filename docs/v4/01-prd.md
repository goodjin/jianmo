- **状态**: 草稿

## 1. 项目概述

### 1.1 背景与目标

**背景**: 当前 Markly v3 基于 Milkdown (ProseMirror) 实现，存在版本碎片化严重、插件生态不稳定、初始化复杂易白屏等问题。需要重构为更稳定、可测试、大模型友好的架构。

**目标**: 基于 CodeMirror 6 构建新一代 Markdown 编辑器，实现：
1. 架构清晰，状态可预测
2. 大模型易于生成正确代码
3. 可测试性强，自动化验证成本低
4. 性能优秀，支持大文档

**对标参考**: 参考 Obsidian 的编辑体验和 Zenn 的 CodeMirror 6 实现，采用装饰模拟 WYSIWYG 方案。

### 1.2 目标用户

| 用户类型 | 描述 | 核心需求 |
|---------|------|---------|
| 技术写作者 | 编写技术文档、博客的开发者 | 高效的 Markdown 编辑、代码高亮、图表支持 |
| 知识管理者 | 使用 Markdown 管理笔记的用户 | 即时渲染、双向同步、大纲导航 |
| VS Code 用户 | 希望在 IDE 内获得优质 Markdown 体验的用户 | 与 VS Code 深度集成、自定义配置 |

### 1.3 核心价值主张

"CodeMirror 6 驱动的稳定、高性能 Markdown 编辑器，提供类 Typora 的即时渲染体验，同时保持纯文本的可控性和可测试性。"

### 2.2 用户故事

#### US-001: 即时渲染编辑

**格式**: 作为技术写作者，我想在输入 Markdown 时立即看到渲染效果，以便专注于内容而非语法。

**详细描述**:
- **角色**: 技术写作者
- **功能**: 输入 `# Title` 后立即显示为标题样式
- **价值**: 减少认知负担，提高写作流畅度
- **对标参考**: Obsidian 的 Live Preview 模式

**验收标准**:
- AC-001-01: 输入 `# ` 后，后续文本立即显示为 H1 样式
- AC-001-02: 输入 `**bold**` 后，文本立即显示为粗体
- AC-001-03: 输入 `- [ ] task` 后，显示为可点击的复选框
- AC-001-04: 光标离开后，Markdown 标记符隐藏（如 `#`、`*`）
- AC-001-05: 光标进入已渲染元素时，显示原始 Markdown 标记

**边界条件**:
- 输入: 标准 Markdown 语法
- 输出: CodeMirror 6 装饰渲染的视觉效果
- 异常: 非法 Markdown 语法保持原文显示

**交互细节**:
- **编辑状态指示**: 光标进入已渲染元素时，显示淡色 Markdown 标记作为提示（如 `# ` 显示为灰色，标题文本显示为黑色）
- **模式指示器**: 编辑器右上角显示当前模式图标（IR/Source/Split）
- **过渡动画**: 标记符显示/隐藏使用 150ms 的淡入淡出动画
- **键盘触发**: 按 `Ctrl/Cmd + /` 可临时切换当前行的编辑/渲染状态

### 2.4 业务流程

#### Flow-001: 模式切换

**流程描述**: 用户在三种编辑模式之间切换

**状态定义**:
| 状态 | 描述 | 可转换到 |
|-----|------|---------|
| IR_MODE | 即时渲染模式（Instant Rendering） | SOURCE_MODE, SPLIT_MODE |
| SOURCE_MODE | 源码模式 | IR_MODE, SPLIT_MODE |
| SPLIT_MODE | 分屏模式 | IR_MODE, SOURCE_MODE |

**状态转换**:
| 当前状态 | 触发事件 | 下一状态 | 条件 |
|---------|---------|---------|------|
| IR_MODE | toggleMode | SOURCE_MODE | 无 |
| IR_MODE | toggleMode | SPLIT_MODE | 无 |
| SOURCE_MODE | toggleMode | IR_MODE | 无 |
| SPLIT_MODE | toggleMode | IR_MODE | 无 |

**切换动画**:
- 模式切换时使用 200ms 的淡入淡出过渡
- 切换前保存当前光标位置，切换后恢复
- 切换过程中显示加载指示器（防止白屏）

**流程图**:
```
[IR_MODE] ←→ [SOURCE_MODE]
    ↑↓
[SPLIT_MODE]
```

#### Flow-002: 图片粘贴处理

**流程描述**: 用户粘贴图片时的处理流程

**步骤**:
1. 监听粘贴事件（`Ctrl/Cmd + V` 或拖拽）
2. 检查剪贴板内容类型
3. 如果是图片，读取为 Blob
4. 生成唯一文件名（格式：`image-{timestamp}-{random}.{ext}`）
5. 检查磁盘空间和写入权限
6. 保存到配置的 assets 目录
7. 在光标位置插入 Markdown 图片语法
8. 发送消息到 VS Code 扩展保存文件

**异常处理流程**:
| 异常类型 | 检测时机 | 处理方式 | 用户提示 |
|---------|---------|---------|---------|
| 磁盘空间不足 | 步骤 5 | 取消保存，保留占位符 | "存储空间不足，请清理后重试" |
| 写入权限不足 | 步骤 5 | 弹出文件选择器，让用户选择新路径 | "无写入权限，请选择其他保存位置" |
| 文件名冲突 | 步骤 4 | 自动添加序号（`image-1.png`） | 无提示，静默处理 |
| 网络/通信失败 | 步骤 8 | 重试 3 次，失败后显示错误占位符 | "保存失败，点击重试" |
| 不支持的图片格式 | 步骤 2 | 取消操作 | "仅支持 PNG、JPG、GIF、WebP 格式" |

## 3. 非功能需求

### 3.1 性能需求

| 指标 | 要求 | 对标参考 |
|-----|------|---------|
| 首屏渲染 | < 500ms | Obsidian < 1s |
| 输入响应 | < 16ms（60fps）| 行业标配 |
| 大文档支持 | 10MB 文件流畅编辑 | CodeMirror 6 虚拟滚动 |
| 内存占用 | < 200MB | 优于 Milkdown |
| 包体积 | < 500KB (gzip) | CodeMirror 6 模块化 |

**大文件处理 UX**:
| 场景 | 处理方式 | 用户提示 |
|-----|---------|---------|
| 文件 > 1MB | 显示加载进度条 | "正在加载大文件..." |
| 文件 > 10MB | 提示性能警告 | "文件较大，部分功能可能受限" |
| 文件 > 50MB | 建议只读模式 | "文件过大，建议以只读模式打开" |
| 懒加载 | 仅渲染可视区域内容 | 滚动时动态加载 |

### 5.3 最佳实践建议

#### 推荐采用

1. **CodeMirror 6 装饰系统实现 WYSIWYG**
   - 来源：Obsidian 的实践证明稳定可行
   - 理由：比 ProseMirror 结构化方案更简单，性能更好

2. **不可变状态管理**
   - 来源：CodeMirror 6 原生设计
   - 理由：状态可预测，易于测试和调试

3. **模块化 Extension 架构**
   - 来源：CodeMirror 6 核心设计
   - 理由：功能可独立开发、测试、复用

#### 建议避免

1. **避免直接操作 DOM**
   - 原因：CodeMirror 6 的 View 层封装了 DOM 操作
   - 替代：使用 ViewPlugin 和 Decoration

2. **避免双向数据绑定**
   - 原因：与 CodeMirror 6 的单向数据流冲突
   - 替代：使用 updateListener 监听变更

3. **避免在 WebView 中使用 Monaco**
   - 原因：VS Code WebView 无法访问 Monaco
   - 替代：使用 CodeMirror 6

#### 创新机会

1. **VS Code 配置双向同步**
   - 竞品痛点：编辑器配置与 VS Code 设置割裂
   - 我们的方案：自动同步主题、字体、缩进等设置

2. **大模型友好的架构**
   - 竞品现状：复杂架构难以 AI 辅助开发
   - 我们的优势：清晰的分层，易于生成正确代码
