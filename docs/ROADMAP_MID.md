# Markly 路线图 · 中期（Mid Phase）

本文档约定 **中期阶段**（建议：**在近期封板后再经历 12–24 个月或约 18–36 个小版本**，视团队吞吐调整）的全面事项与全局顺序。里程碑编号 **`M276–M360`**。

**前置**：近期阶段达到 `docs/ROADMAP_NEAR.md` 的「移交中期」标准。  
**关联**：`docs/IR_REMOVAL_ROADMAP.md`（Phase 2–3）、`docs/PROTOCOL_VERSIONING_PLAN.md`、`docs/MIGRATION_TEMPLATE.md`、`docs/DEFAULT_EDITOR_STRATEGY.md`、`docs/PRIVACY_AND_PERMISSIONS_AUDIT.md`、`docs/M200-2.0-GONOGO.md`。  
**下阶段**：`docs/ROADMAP_FAR.md`（`M361` 起，`M450` 为远期健康度自评参照）。

---

## 1. 阶段定位

- **主旋律**：在 Rich/Source 成熟的前提下 **收缩并移除 IR 实现**；建立 **可重复的大版本决策机制**（含协议版本化预备）；显著提升 **无障碍、国际化、集成测试与跨环境支持**。
- **版本策略**：以 **MINOR** 为主；如出现 breaking，必须走 **`M100/M200` 评审 + `PROTOCOL_VERSIONING_PLAN`**，并准备 **`MIGRATION_TEMPLATE`**——**但不强制本阶段末就发 2.0**。

## 2. 非目标（默认不做，除非单列决策）

- 不默认承诺 **多人实时协作编辑**（可放在远期研究）。
- 不默认承诺 **云端同步/账号体系**。

## 3. 门禁

- 同近期，外加：
- **IR 删除后**：`EditorMode` 类型兼容层策略明确；所有单测与 E2E 去除 IR 路径。
- **Breaking 变更**：必须更新 `messageGuards`、迁移说明与 `CHANGELOG`「Breaking」小节。

## 6. 阶段出口标准（移交远期或进入 2.0 专项）

满足以下之一即可进入 `docs/ROADMAP_FAR.md` 或成立 **2.0 项目组**：

1. **IR 实现删除**已发版且稳定 ≥ N 个迭代。  
2. **协议版本化/迁移机制**已落地或可一键回滚。  
3. **E2E 核心路径**在目标平台上稳定。  
4. **i18n 架构 + A11y 目标**已书面化并有关闭项计划。  
5. 若中期即将 breaking：完成 **`M200-2.0-GONOGO`** 复盘与迁移演练（M340）。
